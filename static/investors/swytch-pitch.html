The Problem

Merchants solved this
in the 14th century.

Venetian traders kept distributed ledgers across a dozen cities with no central authority, no consensus rounds, and no data loss. They did it with a simple rule: every entry records what it saw before it was written. Causality, not clocks. It worked for 600 years.

Then we built computers — and decided to do it differently. Here's what we replaced it with:

  • To write a value, first contact a globally elected leader. If the leader is unreachable, stop. Wait for an election. This is Raft. This is what runs your database.
  • To run multi-region, synchronize atomic clocks accurate to 7 milliseconds using GPS satellites and cesium oscillators, then add deliberate wait periods to ensure the clocks agree. This is Google Spanner's TrueTime. This is the state of the art.
  • To get active-active without a leader, merge concurrent writes using CRDTs — and accept that some operations will silently produce wrong answers depending on order. This is Redis Enterprise at $80k+.
  • Lamport wrote down the simpler path in 1978: track what each write actually saw, not when it happened. Nobody built it. We did.

Swytch Cloud

What you're actually
paying for.

The OSS nearcache is free, single-region, and always will be. Swytch Cloud is what you get when your data needs to survive a region going dark — and keep accepting writes while it does.

Multi-region active-active

Every region accepts reads and writes simultaneously. No primary. No follower lag. A region fails and the others continue without an election, a failover script, or a pagerduty alert.

Durable causal history

Every write is immutable and causally linked. Point-in-time recovery to any moment. You can replay, audit, or restore — not because you planned for it, but because the architecture makes it free.

Exactly-once delivery

When nodes can communicate, conflicting transactions are resolved by deterministic fork-choice — exactly one commits, the other retries. In safe mode (the default), transactions correctly block during a partition and resume cleanly on reconnection. Holographic mode lets designated key prefixes keep accepting transactions during a partition, with automatic merge on reconnection — or human review if both sides committed on the same keys.

Managed operations

No cluster to configure. No replication topology to maintain. No sentinel. You point your client at the local nearcache — already running in-process — and the rest just works. Durability, replication, and multi-region consistency happen behind the scenes.
